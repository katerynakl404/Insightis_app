Shadows & elevation

Every box-shadow flows through a --shadow-* token — component rules read box-shadow:var(--token), never an inline recipe. Three families: elevation (depth at rest / hover / overlay), focus (the focus-visible ring), rim (inset 1px edge for raised toggle / pill surfaces). rgba(0,0,0,α) inside a shadow is the one allowed raw-colour exception.

Usage
Consume the token, never the recipe. A component rule reads the shadow from the token so a future tune is one edit, not N:
.segctrl-btn.is-active{ box-shadow:var(--shadow-rim-active) }  ✓
.segctrl-btn.is-active{ box-shadow:inset 0 0 0 1px var(--border), 0 1px 2px rgba(0,0,0,.06) }  ✗ inline recipe — drifts
Picking elevation (low → high): rest → --shadow-rest; flat row hover → --shadow-card-hover; card that lifts → --shadow-lift-hover; any floating surface — menu, autocomplete, popover, toast → --shadow-overlay; modal dialog → --shadow-modal. Directional drawers (sheet) keep an inline edge-pointing shadow — not a depth rung.
Picking focus: use --shadow-focus for every outset focus ring — the halo colour flows through --focus-ring (brand by default), so a role that needs a different focus tone overrides --focus-ring locally rather than swapping the shadow. --shadow-focus-inset only for an interactive child nested inside a hovering / scrolling parent whose mask would clip an outset ring.
Surface-matched gap (intentional exception): elements sitting on the page bg rather than a card use a --bg gap inline (.chat-row, .dsf-meta-dl) — the focus ring's inner gap matches the surface beneath it, so these stay inline rather than using the --card-gap token.
